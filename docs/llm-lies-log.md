LLM Lies Log
=============
Fabricated, outdated, and unverified claims generated by LLMs and
committed to this repository without verification against live sources.
This file exists for prevarication verification and prevention.

Audited: 2026-03-12 (initial creation — 2 entries from Codex audit)
Repos:   wp-security-style-guide


1. INCORRECT REGEX — Glossary term count excludes non-letter entries
   Files:  docs/current-metrics.md (Architectural Facts table,
           Verification Script)
   Claim:  `grep -cE '^\*\*[A-Z]' WP-Security-Style-Guide.md` counts
           all glossary terms.
   Reality: The `[A-Z]` character class excludes entries that start with
            non-letter characters. The glossary includes entries like
            "**2FA / MFA**" that start with a digit. The original pattern
            returned 128; the correct count is 137 (9 entries missed).
            Correct command: `grep -cE '^\*\*' WP-Security-Style-Guide.md`
   Source:  Running both patterns against the actual document.
   Found:  Codex audit, 2026-03-12. Severity: Medium.
   Fixed:  923b3b3 (Claude Code, same day)
   Notes:  The ai-assisted-docs cross-repo metrics table already had
            the correct value (137) because it was populated from the
            fixed pattern. The error was in the Style Guide's own
            verification file, meaning the "authoritative" source for
            the count was wrong while a downstream copy was correct.

2. STALE PATTERN IN VERIFICATION SCRIPT — Same glossary regex
   Files:  docs/current-metrics.md (Verification Script section,
           echo command at the end of the bash block)
   Claim:  The verification script used `grep -cE '^\*\*[A-Z]'` to
           report the glossary term count.
   Reality: Same bug as #1. The script would report 128 when run,
            contradicting the table above it that (after the table fix)
            correctly said 137. Both locations needed fixing.
   Source:  Same as #1.
   Found:  Codex audit, 2026-03-12. Severity: Medium.
   Fixed:  923b3b3 (Claude Code, same day — replace_all fixed both)


ROOT CAUSE
----------
The glossary regex was written assuming all glossary terms start with an
uppercase letter. This assumption was never tested against the actual
glossary content. The `[A-Z]` character class is a common LLM pattern for
"looks like a defined term" that doesn't account for entries beginning
with digits, symbols, or other non-letter characters.

PREVENTION
----------
1. Every verification command in current-metrics.md MUST be run and its
   output compared to the stated value before committing.
2. The CI workflow (.github/workflows/validate-metrics.yml) automates
   this check on every push and PR that touches the document or metrics.
3. When writing regex patterns, test against edge cases in the actual
   data — not against an imagined "typical" entry.
